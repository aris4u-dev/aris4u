Install

ARIS4U is a Claude Code plugin. The recommended path is 3 commands inside Claude Code — no cloning, no editing files by hand. Onboarding happens inside Claude.

👋 Not a developer? Even simpler.

Install once (below). The first time you open Claude, ARIS4U finishes the setup by itself — it generates your config, adds the recommended settings for you (you just approve), and turns on the semantic router. After that you never touch a command: just describe what you want in plain language and ARIS4U + Claude pick the right tools and do it, from A to Z. You focus on the what; they handle the how.

✅ Native install · 3 commands

Type them inside Claude Code. Skills, agents, hooks and the MCP server are auto-discovered:

/plugin marketplace add https://github.com/aris4u-dev/aris4u.git
/plugin install aris4u@aris4u-dev
/reload-plugins

On the first session, a SessionStart hook builds the Python environment (.venv312 + deps) once, on its own. Then the guided onboarding detects what's missing, generates your config and shows you exactly which env vars to paste (the only manual step):

/aris-onboard     # full onboarding: config + env vars + verification (incl. /aris-status)

You can run /aris-status anytime to re-confirm the state (hooks + MCP + memory alive).

Manual install / development (alternative, step by step)

If you'd rather clone the repo — for example to develop on ARIS4U. ~5 minutes.

Requirements

Python 3.11+ (3.12 recommended), Git, and Claude Code with your Max account. jq is REQUIRED — used by the parallel-dispatch-guard and gpu-crash-guard hooks. Without it, those guards silently fail.

python3 --version   # >= 3.11
git --version
claude --version
1

Clone the repository

cd ~/projects
git clone https://github.com/aris4u-dev/aris4u.git aris4u
cd aris4u
2

Run the installer

Creates the venv, installs dependencies, passes the verification gate and generates your config. Idempotent.

bash install.sh           # interactive: pick your projects
# or fully automatic (auto-detects hardware and projects):
bash install.sh --yes
# additional flags: --cron (installs daily auto-adapt cron) · --no-init (skips config generation)

Generates ~/.aris4u/config.json with your hardware/projects — nothing is hard-wired to the author.

3

Register it as a plugin in Claude Code

This is how Claude loads the hooks and the 7 MCP tools, by plugin convention.

claude plugin marketplace add ~/projects/aris4u
claude plugin install aris4u@aris4u-dev
Heads up: if this machine already had ARIS4U wired by hand in ~/.claude/settings.json, remove those blocks (with a backup) after confirming the plugin loads, or the hooks would run twice.
4

Apply the recommended Claude configuration

The env vars from the Claude config tab tune ARIS4U (cache, verify-gate, memory). Copy them into your settings.

# open your global settings to paste the "env" block (see Config tab)
$EDITOR ~/.claude/settings.json
5

Verify it's alive

Open Claude Code in any project and ask for the status.

/aris-status

Optional · Local semantic memory with Ollama

Without Ollama it works with text search + guards. With Ollama you add semantic recall, the Conductor router and dialectic, all local (private).

# install Ollama: https://ollama.com
ollama pull embeddinggemma       # PRIMARY — session recall (EMBED_MODEL in engine/v16/config.py)
ollama pull mxbai-embed-large    # capability router (Conductor semantic matching)

Both models are needed for full semantic recall + Conductor routing. Without Ollama the system degrades to FTS5 text search — guards still work.

Correct Claude configuration

ARIS4U works out of the box, but these variables in ~/.claude/settings.json tune it: they turn on caching, the code verify-gate, and give memory capture enough time. Paste the "env" block and restart Claude Code.

{
  "env": {
    // — performance and memory —
    "ENABLE_PROMPT_CACHING_1H": "true",
    "CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS": "15000",

    // — ARIS4U: depth, verification, privacy —
    "ARIS4U_DEPTH_PROTOCOL": "1",
    "ARIS4U_CONDUCTOR_ENFORCE": "1",
    "ARIS4U_HEALTHCARE": "0",

    // — optional (fine tuning) —
    "ARIS4U_ROUTER_SEM_THRESHOLD": "0.70",
    "ARIS4U_DIVERSE_RECALL": "0"
  }
}
VariableWhat it does
ENABLE_PROMPT_CACHING_1H
true		 Caches the prompt for 1 hour. Saves tokens and speeds up responses.recommended
CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS
15000		 Gives ARIS4U time to capture the session memory on close (commits, decisions, recall). Without it, capture may get cut off.recommended
ARIS4U_DEPTH_PROTOCOL
1		 Classifies each request and adjusts depth: simple goes straight through; complex is pushed to research and verify.recommended
ARIS4U_CONDUCTOR_ENFORCE
1		 The verify-gate (Pillar 2). If you edited code and didn't run tests/lint, it reminds you when the turn closes — without blocking. Set 0 to turn it off.recommended
ARIS4U_HEALTHCARE
0		 PHI/HIPAA off. Set it to 1 only if you work with medical data (enables extra guards and local routing).default
ARIS4U_ROUTER_SEM_THRESHOLD
0.70		 Threshold of the Conductor's semantic router. Higher = fewer but more precise suggestions. Lower = more coverage.optional
ARIS4U_DIVERSE_RECALL
0 / 1		 At 1, recall returns more diverse (less redundant) results. Useful in large projects.optional
ARIS4U_AUTOPILOT
0 / 1		 For non-developers who want fully hands-off. At 1, when your request matches a known capability, Claude executes it instead of just suggesting. Leave 0 if you're a developer.non-dev
These configure themselves (don't touch): ARIS4U_ROOT, ARIS4U_CONFIG, ARIS4U_LOG_FILE — the plugin derives them from your install. Your hardware, projects and clients live in ~/.aris4u/config.json (generated by aris4u init), not here.
Apply: edit ~/.claude/settings.json and merge the "env" block with whatever you already have, or ask Claude to do it (see the last prompt in the next tab). Restart Claude Code for it to take effect.

What ARIS4U gives you

Claude is brilliant in every conversation but forgets on close and doesn't watch your code. ARIS4U is the layer that gives it memory, verification and the ability to auto-orchestrate its toolkit.

V18 · Intelligent model router

🧭 Tri-model routing doctrine (Fase A)

V18 routes every Agent() to the right model tier by cognitive difficulty, not by habit. Without model= explicit, agents inherit the session thread — Fable, the most expensive model, used for work that doesn't need it.

Fable = session thread (strategy only, not a subagent) · Opus = final synthesis/verdict · Sonnet = the bulk of fan-out (verify, explore, review, edit) · Haiku = trivial (classify, format, label). Engine: tools/model_router.py (route_model, session_model, recall_tier). Every prompt auto-injects 🧭 ROUTING advisory via UserPromptSubmit hook.

💰 Cost & routing observability (Fase C)

New in V18: tools/cost_report.py measures routing discipline — what % of Agent() calls specified model= explicitly, model distribution, and relative cost. Console endpoint: GET /routing.

Run: python3 tools/cost_report.py (last 7 days) or --all --json. Live dashboard: console /routing. Cost is a relative proxy by tier (no per-subagent token counts from hooks) — the actionable signal is the discipline %, not the absolute number.

🧠 Own memory — decoupled from claude-mem (Fase E)

ARIS4U no longer depends on the third-party claude-mem plugin. Observation text lives in its own observations_local table (FTS5-indexed) inside data/sessions.db. Semantic index: data/aris_vectors.db (sqlite-vec, EmbeddingGemma 768d).

Storage: data/sessions.db (decisions · guards · digests · observations_local) + data/aris_vectors.db (semantic). claude-mem is now optional — ARIS4U's memory is self-contained. Outside healthcare, aris_dialectic/aris_structure/aris_critique delegate to Sonnet harness subagents instead of local Ollama (Fase B) — local stays only for PHI data.
The two pillars

🧠 Memory that persists

What you decide today is still there tomorrow, in another session and with another model. Each project has its isolated memory: when you enter a client, Claude brings its decisions and guards.

Auto-captures every session · hybrid text+semantic recall · aris_recall_client, aris_search.

🛡️ Code verification

The other pillar. Layer 1: on edit, it runs ruff/native analyzer + types/tests on commit. Layer 2 — verify-gate: if you touched code and didn't verify, it reminds you when the turn closes — without blocking.

IDE-grade diagnostics per edit · closing reminder with ARIS4U_CONDUCTOR_ENFORCE=1.

🎼 Conductor — auto-orchestrates your toolkit

Claude has the logic; ARIS4U gives it the map. On every non-trivial task it recognizes all your skills/MCP (incl. future ones) and guides you through the UNDERSTAND → DESIGN → BUILD → VERIFY cycle with the right capability.

Live inventory + local semantic router. On the trivial it stays out of the way; on the complex it orchestrates.
Judgment & control

Your standards, enforced automatically

The guards are your own rules, watched automatically: they stop Claude from repeating known mistakes or breaking conventions, before they happen. Some are blocking (migrations, sensitive data); the rest are warnings.

Automatic depth: on the complex, it pushes to research and compare before answering.

Traceability & private

It auto-captures commits and session summaries — you can always reconstruct what was decided and why. The memory, the embeddings and the review run on your machine; optional PHI-safe mode.

All local, no third parties · auditable.

Hardware-aware

It detects your CPU, RAM and GPU (and your workers, if you have them) and distributes work without saturating the machine: parallelizes when there's room, holds back when there isn't.

Generated by aris4u init in your ~/.aris4u/config.json.
The 7 MCP tools · what each one does

You invoke them when you need them (or the Conductor suggests them). Each solves a concrete moment — aris_structure and aris_critique use the optional local body (MLX: mlx-community/Mistral-Small-3.2-24B-Instruct-2506-4bit · Ollama: mxbai-embed-large):

aris_recall_clientmemory · per client

Brings a project's context before you touch it

Recovers the decisions, standards and notes you saved for that client, isolated from the rest.

When: when you start working on a client.
aris_searchmemory · global

Searches your whole history

"What did we decide about X?" — it finds it across all your sessions, by text and by meaning.

When: when you already covered something but don't remember what you concluded.
aris_ingestmemory · write

Saves a decision or a standard

Locks in a conclusion or rule so it survives the session close and reappears when it matters.

When: right after deciding something important.
aris_dialecticquality · code

Three-voice review of your critical code

Builder, reviewer and security examine your code before you commit. V18 (Fase B): outside healthcare, the three roles delegate to Sonnet harness subagents — replacing local Ollama that was failing ~85% of the time. Local stays only for PHI contexts.

When: before merging sensitive code (auth, payments, migrations).
aris_critiquequality · ideas

Finds the weak points of a proposal

Critiques an idea from several angles and flags its risks and fragile assumptions.

When: when you finish designing something, to see what you missed.
aris_structuredesign · ideas

Turns a raw idea into a specification

Takes a vague statement and organizes it (goal, parts, criteria) ready to build.

When: when starting something new and fuzzy.
aris_healthsystem · diagnostics

Checks that everything is alive

Verifies the state of the memory and the local models (Ollama).

When: if semantic recall or dialectic don't respond.

Troubleshooting

Most issues are one command away. ARIS4U is fail-open: if something isn't set up, it degrades gracefully instead of breaking — you can always ask Claude "run /aris-onboard" and it fixes what's missing.

SymptomFix
ARIS4U doesn't seem active / a / command isn't recognized Run /reload-plugins, or restart Claude Code. Then /aris-status to confirm hooks + MCP + memory are alive.
First session says "creating environment…" and is slow One-time build of the Python env (.venv312, ~30–60s). Normal — it only happens once, then every session is instant.
Setup didn't start by itself Just type /aris-onboard. It detects what's missing, generates your config, adds the env vars for you (with your OK) and verifies.
Hooks seem to run twice This machine had ARIS4U wired by hand in ~/.claude/settings.json. Remove those blocks (make a backup first) — the plugin already loads them.
Semantic recall / the router feels weak Install Ollama + ollama pull embeddinggemma (primary recall) + ollama pull mxbai-embed-large (capability router), then build the index (/aris-onboard does it, or python3 tools/capability_semantic.py --build). Without Ollama it falls back to FTS5 text search — still works, just less semantic.
MCP tools don't appear /reload-plugins; confirm the plugin's .venv312 exists (a first session builds it). If needed, run bash install.sh in the repo.
I want it less proactive Set any of these env vars to 0: ARIS4U_AUTOPILOT, ARIS4U_CONDUCTOR_ENFORCE, ARIS4U_AUTO_ONBOARD.
Update: in Claude Code, /plugin → update aris4u (or re-add the marketplace). Uninstall: claude plugin uninstall aris4u; optionally remove ~/.aris4u/ and the ARIS4U env vars from your settings. Your memory lives in data/sessions.db (decisions/guards/observations) and data/aris_vectors.db (semantic index) inside the ARIS4U repo folder — they stay unless you delete them. The old ~/.claude-mem/ is no longer used by ARIS4U V18.
Still stuck? Ask Claude directly, in plain language: "ARIS4U doesn't seem to be working — check /aris-status, tell me what's missing and fix it." That's the whole point — you describe the problem, it handles the rest.

👋 ¿No eres desarrollador? Aún más simple.

Instala una vez (abajo). La primera vez que abras Claude, ARIS4U termina el setup solo — genera tu config, añade la configuración recomendada por ti (tú solo apruebas) y enciende el router semántico. Después no tocas ningún comando: solo describe en lenguaje natural lo que necesitas y ARIS4U + Claude eligen las herramientas correctas y lo hacen, de la A a la Z. Tú te enfocas en el qué; ellos se encargan del cómo.

✅ Instalación nativa · 3 comandos

Escríbelos dentro de Claude Code. Se auto-descubren skills, agents, hooks y el servidor MCP:

/plugin marketplace add https://github.com/aris4u-dev/aris4u.git
/plugin install aris4u@aris4u-dev
/reload-plugins

En la primera sesión, un hook SessionStart crea el entorno Python (.venv312 + deps) una sola vez, solo. Luego el onboarding guiado detecta lo que falte, genera tu config y te muestra exactamente qué env vars pegar (el único paso manual):

/aris-onboard     # onboarding completo: config + env vars + verificación (incl. /aris-status)

Puedes correr /aris-status cuando quieras para reconfirmar el estado (hooks + MCP + memoria vivos).

Instalación manual / desarrollo (alternativa, paso a paso)

Si prefieres clonar el repo — por ejemplo para desarrollar sobre ARIS4U. ~5 minutos.

Crea el venv, instala dependencias, pasa el gate de verificación y genera tu config. Idempotente.

Opcional · Memoria semántica local con Ollama

Sin Ollama funciona con búsqueda por texto + guards. Con Ollama añades recall semántico, router del Conductor y dialéctica, todo local (privado).

# instala Ollama: https://ollama.com
ollama pull embeddinggemma       # PRIMARIO — recall de sesiones (EMBED_MODEL en engine/v16/config.py)
ollama pull mxbai-embed-large    # router de capacidades (matching semántico del Conductor)

Se necesitan ambos modelos para recall semántico completo + router del Conductor. Sin Ollama el sistema degrada a búsqueda FTS5 — los guards siguen funcionando.

Configuración de Claude correcta

ARIS4U funciona out-of-the-box, pero estas variables en ~/.claude/settings.json lo afinan: activan el cache, el verify-gate de código, y dan tiempo a la captura de memoria. Pega el bloque "env" y reinicia Claude Code.

{
  "env": {
    // — rendimiento y memoria —
    "ENABLE_PROMPT_CACHING_1H": "true",
    "CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS": "15000",

    // — ARIS4U: profundidad, verificación, privacidad —
    "ARIS4U_DEPTH_PROTOCOL": "1",
    "ARIS4U_CONDUCTOR_ENFORCE": "1",
    "ARIS4U_HEALTHCARE": "0",

    // — opcionales (ajuste fino) —
    "ARIS4U_ROUTER_SEM_THRESHOLD": "0.70",
    "ARIS4U_DIVERSE_RECALL": "0"
  }
}
VariableQué hace
ENABLE_PROMPT_CACHING_1H
true		 Cachea el prompt 1 hora. Ahorra tokens y acelera las respuestas.recomendado
CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS
15000		 Da tiempo a que ARIS4U capture la memoria de la sesión al cerrar (commits, decisiones, recall). Sin esto, la captura puede cortarse.recomendado
ARIS4U_DEPTH_PROTOCOL
1		 Clasifica cada petición y ajusta la profundidad: lo simple va directo; lo complejo empuja a investigar y verificar.recomendado
ARIS4U_CONDUCTOR_ENFORCE
1		 El verify-gate (Pilar 2). Si editaste código y no corriste tests/lint, al cerrar el turno te lo recuerda — sin bloquear. Pon 0 para apagarlo.recomendado
ARIS4U_HEALTHCARE
0		 PHI/HIPAA apagado. Ponlo a 1 solo si trabajas con datos médicos (activa guards extra y routing local).por defecto
ARIS4U_ROUTER_SEM_THRESHOLD
0.70		 Umbral del router semántico del Conductor. Más alto = menos sugerencias pero más precisas. Menos = más cobertura.opcional
ARIS4U_DIVERSE_RECALL
0 / 1		 A 1, el recall trae resultados más diversos (menos redundantes). Útil en proyectos grandes.opcional
ARIS4U_AUTOPILOT
0 / 1		 Para no-desarrolladores que quieren todo hands-off. En 1, cuando tu petición corresponde a una capacidad conocida, Claude la ejecuta en vez de solo sugerirla. Déjalo en 0 si eres desarrollador.no-dev
Se configuran solas (no las toques): ARIS4U_ROOT, ARIS4U_CONFIG, ARIS4U_LOG_FILE — el plugin las deriva de tu instalación. Tu hardware, proyectos y clientes viven en ~/.aris4u/config.json (lo generó aris4u init), no aquí.
Aplicar: edita ~/.claude/settings.json y fusiona el bloque "env" con el que ya tengas, o pídeselo a Claude (ver el último prompt del tab siguiente). Reinicia Claude Code para que tome efecto.

✅ Lo más simple — un solo comando

Tras instalar el plugin (tab Instalación), este comando hace los 6 pasos de abajo por ti, dentro de Claude: detecta lo que falta, genera tu config, te muestra exactamente qué env vars pegar y verifica el estado.

/aris-onboard

¿Prefieres controlar cada paso? Estos son los mismos 6, como prompts manuales que pegas en orden:

📦

Claude clona el repo, corre el instalador y te avisa si algo falla.

Pega en Claude Code Clona el repositorio https://github.com/aris4u-dev/aris4u.git dentro de ~/projects (como directorio "aris4u"), entra a la carpeta y ejecuta `bash install.sh`. Explícame cada paso, muéstrame la salida del gate de verificación, y si algo falla diagnostícalo antes de continuar. No modifiques mi ~/.claude/settings.json sin avisarme.

2 · Configurar para mis proyectos

Detecta tu hardware real y registra tus proyectos/clientes.

Pega en Claude Code Corre `python3 tools/aris4u_init.py` para generar mi configuración. Antes, escanea ~/projects y muéstrame los proyectos detectados; ayúdame a decidir cuáles son labs y cuáles clientes. Confírmame el hardware (CPU, RAM, GPU) y enséñame el ~/.aris4u/config.json final antes de darlo por bueno.

Qué te aporta ARIS4U

Claude es brillante en cada conversación pero olvida al cerrar y no vigila tu código. ARIS4U es la capa que le da memoria, verificación y la capacidad de auto-orquestar su toolkit.

V18 · Router de inteligencia consciente del modelo

🧭 Doctrina tri-modelo (Fase A)

V18 enruta cada Agent() al tier de modelo correcto por dificultad cognitiva, no por inercia. Sin model= explícito los agentes heredan el hilo — Fable, el modelo más caro, para trabajo que no lo necesita.

Fable = hilo de sesión (estrategia, no subagente) · Opus = síntesis/veredicto final · Sonnet = el grueso del fan-out (verificar, explorar, revisar, editar) · Haiku = trivial (clasificar, formatear, etiquetar). Motor: tools/model_router.py (route_model, session_model, recall_tier). Cada prompt auto-inyecta el aviso 🧭 ROUTING vía hook UserPromptSubmit.

💰 Observabilidad de costo/routing (Fase C)

Nuevo en V18: tools/cost_report.py mide la disciplina de routing — qué % de llamadas Agent() especificó model= explícito, distribución por modelo y costo relativo. Endpoint de consola: GET /routing.

Corre: python3 tools/cost_report.py (últimos 7 días) o --all --json. Dashboard vivo: consola /routing. El costo es una proxy relativa por tier (sin token counts por subagente desde los hooks) — la señal útil es el % de disciplina, no el número absoluto.

🧠 Memoria propia — desacoplada de claude-mem (Fase E)

ARIS4U ya no depende del plugin de terceros claude-mem. El texto de las observaciones vive en su propia tabla observations_local (con índice FTS5) dentro de data/sessions.db. Índice semántico: data/aris_vectors.db (sqlite-vec, EmbeddingGemma 768d).

Almacenamiento: data/sessions.db (decisiones · guards · digests · observations_local) + data/aris_vectors.db (semántico). claude-mem es ahora opcional — la memoria de ARIS4U es autónoma. Fuera de healthcare, aris_dialectic/aris_structure/aris_critique delegan en subagentes Sonnet del harness (Fase B) — local queda solo para datos PHI.
Los dos pilares

🧠 Memoria que persiste

Lo que decides hoy sigue ahí mañana, en otra sesión y con otro modelo. Cada proyecto tiene su memoria aislada: al entrar a un cliente, Claude trae sus decisiones y guards.

Captura sola en cada sesión · recall híbrido texto+semántico · aris_recall_client, aris_search.

🛡️ Verificación de código

El otro pilar. Capa 1: al editar, corre ruff/analizador nativo + tipos/tests en el commit. Capa 2 — verify-gate: si tocaste código y no verificaste, al cerrar el turno te lo recuerda — sin bloquear.

Diagnósticos nivel-IDE por edición · recordatorio de cierre con ARIS4U_CONDUCTOR_ENFORCE=1.

🎼 Conductor — auto-orquesta tu toolkit

Claude tiene la lógica; ARIS4U le da el mapa. En cada tarea no-trivial reconoce todas tus skills/MCP (incl. futuras) y te guía por el ciclo ENTENDER → DISEÑAR → CONSTRUIR → VERIFICAR con la capacidad correcta.

Inventario vivo + router semántico local. En lo trivial no estorba; en lo complejo orquesta.
Criterio & control

Tus estándares, aplicados solos

Los guards son reglas tuyas que se vigilan automáticamente: evitan que Claude repita errores conocidos o rompa convenciones, antes de que ocurran. Algunos bloqueantes (migraciones, datos sensibles); el resto, avisos.

Profundidad automática: en lo complejo, empuja a investigar y comparar antes de responder.

Trazabilidad & privado

Captura sola commits y resúmenes de sesión — siempre puedes reconstruir qué se decidió y por qué. La memoria, los embeddings y la revisión corren en tu máquina; modo PHI-safe opcional.

Todo local, sin terceros · auditable.

Consciente de tu hardware

Detecta tu CPU, RAM y GPU (y tus workers, si tienes) y reparte el trabajo sin saturar la máquina: paraleliza cuando hay margen, se contiene cuando no.

Generado por aris4u init en tu ~/.aris4u/config.json.
Las 7 herramientas MCP · qué hace cada una

Las invocas cuando las necesitas (o el Conductor te las sugiere). Cada una resuelve un momento concreto — aris_structure y aris_critique usan el cuerpo local opcional (MLX: mlx-community/Mistral-Small-3.2-24B-Instruct-2506-4bit · Ollama: mxbai-embed-large):

aris_recall_clientmemoria · por cliente

Trae el contexto de un proyecto antes de tocarlo

Recupera las decisiones, estándares y notas que guardaste para ese cliente, aislado del resto.

Cuándo: al empezar a trabajar en un cliente.
aris_searchmemoria · global

Busca en todo tu historial

"¿Qué decidimos sobre X?" — lo encuentra entre todas tus sesiones por texto y por significado.

Cuándo: cuando ya trataste algo pero no recuerdas qué concluiste.
aris_ingestmemoria · escribir

Guarda una decisión o un estándar

Fija una conclusión o regla para que sobreviva al cierre de la sesión y reaparezca cuando importe.

Cuándo: justo después de decidir algo importante.
aris_dialecticcalidad · código

Revisión a tres voces de tu código crítico

Constructor, revisor y seguridad examinan tu código antes de comprometerlo. V18 (Fase B): fuera de healthcare, los tres roles delegan en subagentes Sonnet del harness — reemplaza el Ollama local que fallaba ~85% del tiempo. Local queda solo para contextos PHI.

Cuándo: antes de mergear código sensible (auth, pagos, migraciones).
aris_critiquecalidad · ideas

Encuentra los puntos débiles de una propuesta

Critica una idea desde varios ángulos y marca sus riesgos y supuestos frágiles.

Cuándo: al terminar de diseñar algo, para ver qué se te escapó.
aris_structurediseño · ideas

Convierte una idea cruda en una especificación

Toma un planteamiento vago y lo organiza (objetivo, partes, criterios) listo para construir.

Cuándo: al arrancar algo nuevo y difuso.
aris_healthsistema · diagnóstico

Chequea que todo esté vivo

Verifica el estado de la memoria y de los modelos locales (Ollama).

Cuándo: si el recall semántico o la dialéctica no responden.

/aris-memory-auditmantener memoria

Mantén tu memoria sana

Revisa lo que ARIS4U recuerda por cliente y detecta contradicciones, datos viejos y huecos.

Cuándo: cada cierto tiempo, para que no acumule decisiones obsoletas.
En una frase: Claude piensa; ARIS4U recuerda, verifica y orquesta. Los gates y la memoria trabajan solos; las herramientas MCP y las skills las usas (o el Conductor te las sugiere) cuando hace falta. Dejas de empezar de cero — y de cerrar código sin verificar.

Solución de problemas

La mayoría se resuelve con un comando. ARIS4U es fail-open: si algo no está configurado, degrada limpio en vez de romperse — siempre puedes pedirle a Claude "corre /aris-onboard" y arregla lo que falte.
